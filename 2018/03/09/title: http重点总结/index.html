http重点总结

Posted on 2018-03-09 | In 网络基础 , http重点总结 | 浏览 次

1、http的主要特点

  • 简单快速：给定url地址返回相应的资源
  • 灵活：同一个http 协议可以返回不同的资源
  • 无连接：每次请求和响应完成后http自动断开连接
  • 无状态：http不保存请求和响应之间的状态，也就是说同一个用户请求两次，http无法识别这两次请求是否是同一个发送的。

2、http报文的组成

请求报文：

  • 请求行：请求方法、页面地址、请求协议和版本号
  • 请求头：
    （1）以key-value形式组成的内容，
    （2）典型的请求头有：user-agent浏览器类型
    （3）Accept：客户端可识别的内容类型列表
    （4）host请求的主机名
  • 空行
  • 请求体

响应报文

  • 状态行：HTTP协议的版本，服务器发回的响应状态代码，状态代码的文本描述
  • 响应头
  • 空行
  • 响应体

htpp的方法

  • GET：用于获取资源
  • POST：用于传输资源
  • PUT： 更新资源或者上传文件
  • DELETE：删除资源
  • HEAD：获取报文首部

GET和POST的区别

  • GET回退不会重复提交请求，而POST则会重复提交请求
  • GET产生的url可以被浏览器收藏，POST产生的URL不会被收藏
  • GET请求就被浏览器主动缓存，而POST则不会
  • GET请求只能通过URL编码，而POST支持多种编码
  • GET请求的参数会被完整的保存在浏览记录里，而POSt不会
  • GET请求在URL中传送的参数是有限制的，POST则没有限制
  • GET与POSt相比更不安全，因为参数直接暴露在URL中而POST放在body中

    http状态码

  • 200:请求成功
  • 204 服务端请求成功但是没有数据
  • 206:客户端发送一个带有range的请求，服务端完成此次请求
  • 301:永久性重定向，将所有的页面转移至新的URL中
  • 302:暂时性重定向，将所有的页面暂时转移至新的URL
  • 304:客户端有缓存文档并发送一个条件性的请求，服务端告诉客户端原来的缓存文档还能继续使用
  • 400:客户端有语法错误，服务端无法解析
  • 401:请求未经授权
  • 403:请求的页面被禁止
  • 404:请求的资源不存在
  • 500:服务器发生不可预期的错误，原来缓存的文档可继续使用
  • 503:服务器发生当机或者临时过载，经过一段时间可继续使用

TCP三次握手和四次挥手

建立连接

  • 首先客户端向服务器发送一个带有syn的数据包、seq等于x的序号列
  • 服务器收到后返回一个syn的数据包，并将ack设置为x+1表示下一个数据包的序号列，再发送一个确认包（ACK）和seq等于y的序号列
  • 客户端收到后发送一个确认包，ack设置为上一个seq的值加一，此时建立起连接

    关闭连接

  • 客户端发送一个fin包表示要关闭客户端与服务器端的连接
  • 服务器端接收以后，发送一个确认包并将确认序列加一
  • 服务器端发送一个fin的包表示要关闭服务器端和客户端的连接
  • 客户端收到后，发送一个带有ack的确认包，

http持久连接

这是http协议1.1版本支持的功能，所谓持久连接是为了解决http协议无连接的问题，也就是每次请求/应答都需要建立新的连接。在持久连接的状态下，客户端和服务端的连接保持有效，避免了每次重复建立新的连接。

http管线化

  • 在持久连接的前提下，http请求和响应的模式为发送一个请求，响应一个请求，发送一个新的请求，响应一个新的请求
  • 管线化是在持久连接的前提下，将发送的请求进行打包，并把响应打包返回的过程
  • 需要注意的是，管线化仅http1.1版本才支持的。并且只有GET和HEAD支持，POST不支持。初次创建连接的时候不应该启动管线机制，因为无法确定服务器是否支持http1.1版本的协议

http2新特性

  • 二进制分祯：将消息分割成更小的消息和祯并对他们采用二进制编码，增加传输性能
  • 压缩头部：使用首部表跟踪存储发送的键值对，对于相同的请求不再重复发送。
  • 多路复用：多路复用允许单一的http2连接发送多重请求
  • 请求优先级：根据不同的优先级确定不同的优先策略
  • 服务器提示：服务端不发送任何资源实体，只发送资源的url，客户端接到提示后会验证缓存请求，当客户端检测真正需要该资源时，则正式发起请求。

cookie、localStorage、sessionStorage的区别和共同点

  • 共同点：都是保存在浏览器端的并且都是同源的
  • 不同点：cookie的数据始终在同源的http请求中携带，而localStorage和sessionStorage不会把数据发到服务器上。仅存储在本地，cookie有路径的概念，并且cookie的存储空间比较小
  • localStorage永久存储在本地，存储的空间比较大，在所有同源的窗口内都是共享的
  • sessionStorage仅存在于同一浏览器打开的页面中，属于会话级别的数据存储

cookie的作用

  • 保护用户的跟踪状态
  • 跟踪用户的行为

cookie和session的区别

  • cookie存放在客户端，session存放在服务端
  • cookie很不安全，session相对安全
  • session存放在服务器上，当服务器访问量增多时比较占用服务器的性能。
  • session的运行依赖session id，而session id存储在cookie中，如果禁用的cookie，session也会失效

什么是csrf？怎么防御csrf？

csrf是让用户在不知情的情况下,冒用其身份发起了一个请求
防御csrf：
1.添加验证码进行防御
2.验证请求中的referer字段
3.请求中加入随机token进行验证
4.尽量使用较为安全的post

Tcp为什么要采用三次握手？两次不行吗？

假设现在a向b发送消息，由于网络或者其他原因导致发送滞留了一段时间，经过延误后发送到b，但这时报文段已经失效。
b接收到失效的报文段之后，误认为a向自己发送链接请求。于是向a发送确认报文段。
由于a并没有发送建立连接的请求，于是对于b发送的确认请求并不会理睬，但是对于b来说会认为成功建立连接，一直等待a发送的数据，从而导致资源的浪费
如果是三次握手的情况下，b收不到确认数据包就不会认为a要建立连接。

为什么在a在time-wait的情况下必须等待2msl的时间呢？

第一为了保证b能收到a发送的报文段，在发送过程中，报文段可能会丢失，超过2msl后b会重新发送请求，接着a会在time-wait中收到b发送的请求，重新回传报文段。
如果不等待2msl，time-wait之后自动关闭，会导致b重发的请求无法接收。
第二，a发送完全部报文段之后，再经过2msl就可以使本连接持续的时间内产生的报文段都在网络中消失，这样就可以使下一个新的连接中不会出现旧连接的报文段。

什么情况下会使用option请求？

1.可以使用 OPTIONS 方法对服务器发起请求，以检测服务器支持哪些 HTTP 方法
2.在 CORS 中，可以使用 OPTIONS 方法发起一个预检请求，以检测实际请求是否可以被服务器所接受。预检请求报文中的 Access-Control-Request-Method 首部字段告知服务器实际请求所使用的 HTTP 方法；Access-Control-Request-Headers 首部字段告知服务器实际请求所携带的自定义首部字段。服务器基于从预检请求获得的信息来判断，是否接受接下来的实际请求。

tcp如何保证稳定性？

1.校验和
2.序列号（到达的序列号）
3.确认应答机制
4.超时重传机制
5.流量控制
6.流量控制
7.拥塞控制：
1）TCP连接初始化，将拥塞窗口设置为1
2）执行慢开始算法，cwind按指数规律增长，直到cwind == ssthress开始执行拥塞避免算法，cwnd按线性规律增长
3）当网络发生拥塞，把ssthresh值更新为拥塞前ssthresh值的一半，cwnd重新设置为1，按照步骤（2）执行。
